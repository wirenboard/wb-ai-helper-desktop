# controller-update

Обновление пакетов контроллера WB и переход между релизами (suite switching) через `apt` и `wb-release -t`. Подгружай на ЛЮБОЕ упоминание обновления софта на контроллере: «обнови контроллер», «apt upgrade», «обнови пакеты», «поставь обновления», «есть ли обновления», «перейди на testing», «вернуться на stable», «сменить релиз», «накатить новый wb-релиз», «dist-upgrade», «обновить wb-rules», «обновить wb-mqtt-serial». Про прошивку по FIT — **не этот скилл**, FIT-прошивку мы вообще не запускаем (см. системный промпт).

**ВАЖНО: прочитай ВЕСЬ скилл перед действиями. НЕ вызывай ssh_exec/ssh_exec_async параллельно с load_skill — сначала прочти инструкции.**

Область: **apt-обновление** (`apt update` / `apt upgrade`) и **переключение релиза** (`wb-release -t <suite>`). Оба сценария трогают rootfs, перезапускают сервисы и при неудаче могут положить контроллер в состояние, где часть пакетов в одном релизе, часть в другом. Перед любым запуском — бэкап через скилл `controller-backup`.

**ssh_exec vs ssh_exec_async — простое правило:**
- Моментальные команды (локальный кеш, без сети): `ssh_exec` — `wb-release`, `dpkg -l`, `apt list`, `apt policy`, `df`, `uptime`, `systemctl status`.
- Сетевые или долгие: `ssh_exec_async` — `apt update` (тянет индексы из сети), `apt install/upgrade/remove` (скачивает + ставит), `wb-release -t` (меняет repos + update + upgrade).
Сервер принудительно заблокирует сетевые apt-команды через `ssh_exec` и автоматически добавит `DEBIAN_FRONTEND=noninteractive` + `-y` (для install/upgrade/dist-upgrade/remove/purge) в `ssh_exec_async`. То есть `apt-get install pkg` через `ssh_exec_async` де-факто выполнится как `DEBIAN_FRONTEND=noninteractive apt-get install -y pkg`. **Не нужно писать `-y` руками** — но если случайно прописал, сервер не дублирует. Исторический кейс: модель запускала `apt-get install` без `-y`, dpkg ждал ответа Y/N, default был N, пакет не обновлялся (реально воспроизведено на A25NDEMJ — wb-mqtt-serial 2.146 vs 2.224 в репе). После `ssh_exec_async` — жди результата, сервер подтолкнёт тебя автоматически.

## Что есть что

- **Пакеты (apt)** — `wb-rules`, `wb-mqtt-serial`, `wb-mqtt-confed` и прочие wb-\* + стандартные Debian-пакеты. Версии в пределах одного релиза. Обновляются штатно через `apt upgrade`.
- **Релиз (suite)** — кодовое имя ветки репозиториев, под которые настроены `/etc/apt/sources.list*`. Пример: `wb-2504` (stable, на апрель 2025). Меняется через `wb-release -t`. Смена релиза — это фактически массовый upgrade пакетов из новых репозиториев + возможная смена версии Debian под капотом (bullseye → bookworm и т.п.).
- **FIT-прошивка** — образ rootfs, заливается через заводскую кнопку или `wb-fw-update`. Перезаписывает /. В этом скилле НЕ рассматривается, модель её не запускает.

Граница: `apt upgrade` в рамках текущего релиза безопасен, `wb-release -t` меняет целую ветку — это крупное изменение.

## Зонтичное согласие (важно!)

Если пользователь дал явное **общее** согласие на весь сценарий — «обнови всё», «сделай это и всё остальное», «доведи до конца», «не спрашивай каждый раз», «мне нужна самая свежая система», «делай», «продолжай в том же духе» — это **разрешение на ВСЕ шаги сценария A** (включая `dist-upgrade` для kept back, рестарты сервисов и перезагрузку после обновления ядра). Не дёргай пользователя «можно?» перед каждым следующим шагом — это раздражает и противоречит сказанному. Просто делай и сообщай результат коротко.

Что зонтичное согласие НЕ покрывает (всегда HITL отдельным подтверждением):
- Смена релиза `wb-release -t` (сценарий B) — это крупное изменение ветки, не часть «обновить всё».
- `wb-release -p`/`-r` (сценарий D) — восстановительные операции.
- Любые destructive команды вне сценария A (rm -rf, форматирование, сброс конфигов).

Если зонтичного согласия не было — следуй HITL-разметке шагов как написано.

## Recon: что сейчас на контроллере

Перед любым действием — recon в два шага:

**Шаг 1** — ОБЯЗАТЕЛЬНО через `ssh_exec` (не `get_controller`! сервер автоматически приложит информацию о новых релизах к выводу `wb-release`):

```bash
wb-release 2>&1 | head -20
echo ===
df -h / /mnt/data | awk 'NR==1 || /\/$|\/mnt\/data/'
echo ===
uptime
```

**Шаг 2** — проверка обновлений через `ssh_exec_async` (apt update может быть медленным, сервер заблокирует его через ssh_exec):

```bash
apt-get update -qq 2>&1 | tail -20
echo ===
apt list --upgradable 2>/dev/null
```

Не обрезай вывод `apt list` — wb-пакеты идут в конце алфавита, `head` их отсечёт.

Дождись завершения через auto-polling (сервер подтолкнёт автоматически).

Что смотреть:
- **Текущий релиз** (`wb-release`) — модель и версия suite.
- **Новые релизы — сервер подскажет автоматически.** Когда ты вызываешь `ssh_exec` с `wb-release` в команде, сервер автоматически сравнит текущий релиз контроллера с последним stable на GitHub и приложит к ответу список новых релизов со ссылками на changelog. Если в ответе есть блок «Доступные обновления релизов»:
  1. **Явно скажи пользователю** про новый релиз.
  2. **Прочитай changelog** последнего релиза (ссылка из ответа) через `web_fetch` и **компактно, по-человечески** расскажи что нового — фокус на пользе: новые устройства, исправления багов, улучшения UI. Не вываливай сырой changelog.
  3. Обновление на новый stable происходит через `apt upgrade` + `apt dist-upgrade` при необходимости (сценарий A), НЕ через `wb-release -t`. Команда `wb-release -t` нужна ТОЛЬКО для переключения stable↔testing.
- **`apt update`** — отработал чисто? Ошибки GPG/404 означают битые источники, дальше идти нельзя. С `-qq` при успехе вывод может быть пустым — это нормально.
- **`apt list --upgradable`** — формат вывода: `пакет/репо версия arch [upgradable from: старая_версия]`. Первая строка `Listing...` — заголовок, не данные. Если после `Listing...` идут строки с пакетами — значит обновления ЕСТЬ, посчитай их. Если только `Listing...` и больше ничего — обновлять нечего. Обрати внимание на wb-\* пакеты вперемежку с debian core.
- **Свободное место на `/`** — если <200 МБ, apt не закончит распаковку. Почисти сперва: `apt clean; journalctl --vacuum-time=7d`.
- **Uptime** — если контроллер только что поднялся, подожди 1-2 минуты, пусть rsyslog/wb-* стабилизируются.

## Clarify: что именно делаем

В зависимости от запроса пользователя — разные сценарии. Не путай их, и не склеивай в один.

### Сценарий A: обновление пакетов

Триггеры: «обнови пакеты», «apt upgrade», «есть обновления?», «обнови контроллер».

Что делать:
1. `save_state_for_diff(sn)` — снимок ДО.
2. Совет «собрать бэкап через `controller-backup`» — особенно если в `apt list --upgradable` есть критичные пакеты (`wb-mqtt-serial`, `wb-rules`). Для мелких обновлений (утилиты, локали) бэкап избыточен — решай с пользователем.
3. Покажи пользователю список upgradable. **HITL: жди подтверждения.**
4. `ssh_exec_async(sn, "apt-get -y upgrade", label="apt upgrade")`.
5. `job_tail` до `state: "exited"`. Если `exitCode !== 0` — разбирай по логу, не скрывай от пользователя.
6. **Проверь kept back.** Если в выводе есть «kept back» или «not upgraded» — значит часть пакетов не обновилась из-за новых зависимостей. Сообщи пользователю какие именно пакеты остались и предложи `apt-get -y dist-upgrade` для полного обновления. Покажи что будет доустановлено/удалено (`apt-get -s dist-upgrade | grep -E '^(Inst|Remv)'`). **HITL: жди подтверждения.**
7. После: `diff_snapshot` чтобы показать что поменялось (версии пакетов, новые сервисы).
8. **Проверь обновлялось ли ядро** — ищи `linux-image` в выводе apt. Если ядро обновилось — объясни пользователю: «Обновилось ядро Linux. Новое ядро заработает только после перезагрузки. До перезагрузки контроллер работает на старом ядре, а новые модули и драйверы могут быть несовместимы — это может вызывать сбои. Рекомендую перезагрузиться сейчас.» **HITL: жди подтверждения**, затем `ssh_exec(sn, "systemctl reboot")`. SSH отвалится на 30-60 секунд.
9. Если в списке upgradable были пакеты, требующие рестарта сервиса (`wb-rules`, `wb-mqtt-*`), проверь: `systemctl is-active <unit>`; если перезапустился сам — ок, если нет — `systemctl restart <unit>` с подтверждением пользователя.

**Отдельный случай — `dist-upgrade`.** Если `apt-get -y upgrade` не тянет один из пакетов из-за конфликта зависимостей (`The following packages have been kept back`), можно попробовать `apt-get -y dist-upgrade`. Но **HITL обязательно**: dist-upgrade может ставить новые пакеты и удалять старые, это риск. Покажи diff пользователю (`apt-get -s dist-upgrade | grep -E '^(Inst|Remv)'`), жди подтверждения.

### Сценарий B: переключение ветки stable↔testing через `wb-release -t <suite>`

Триггеры: «перейди на testing», «вернуться на stable», «хочу testing-ветку», «сменить suite на testing/stable».

**ВАЖНО:** этот сценарий — ТОЛЬКО для переключения между ветками (stable↔testing). Обновление на новый stable-релиз (wb-2504→wb-2602) делается через `apt upgrade` + `dist-upgrade` (сценарий A). Не предлагай `wb-release -t` для обновления версии внутри stable.

Что делать:
1. Узнай **точное** имя целевого релиза — оно меняется со временем. `web_fetch(https://wirenboard.com/wiki/WB_Software_Releases)` или спроси пользователя. Не угадывай: `wb-release -t` не проверяет что такой suite существует до старта, упадёт в середине процесса.
2. `save_state_for_diff(sn)` + **обязательный** `controller-backup`. Смена релиза может сломать кастомные пакеты, конфиги с modified-флагом, конфиги старых версий драйверов.
3. Покажи пользователю: текущий релиз, целевой релиз, примерный diff изменений (или хотя бы размер — `apt-get -s dist-upgrade` после временной смены sources.list неудобен, лучше просто предупреди что обновится сотня пакетов).
4. **HITL.** Получи явное «да», повтори какой будет целевой релиз.
5. `ssh_exec_async(sn, "wb-release -y -t <target>", label="wb-release -t <target>")`. Без `-y` процесс ждёт stdin и повиснет. `--no-preliminary-update` НЕ используй — дефолт правильный.
6. `job_tail` с интервалами, но учти — в ходе перехода SSH может временно отвалиться (рестарт сети/dropbear). Жди `state: "exited"`, если `job_status` возвращает ошибку SSH — подожди минуту и повтори. Процесс продолжает идти на контроллере (`systemd-run --collect`), даже если мы его временно не видим.
7. После `state: "exited"`: `wb-release` снова — должен показать новый suite. `apt list --upgradable` — должен быть пустым. Если висят «kept back» пакеты — сценарий A `dist-upgrade`.
8. `diff_snapshot` против слепка ДО. Разбери с пользователем крупные изменения.
9. Если перезагрузка контроллера не происходит автоматически — скажи пользователю: «рекомендую перезагрузить после смены релиза», спроси разрешения, при согласии — `ssh_exec(sn, "systemctl reboot")`. После перезагрузки SSH временно отвалится на ~30-60 секунд.

`wb-release -p` (`--reset-packages`) и `wb-release -r` (`--regenerate`) — для восстановительных сценариев, а не обычного обновления. Используй только если понимаешь зачем и подтвердил с пользователем.

### Сценарий C: «есть ли обновления» без немедленного действия

Триггеры: «что там по обновлениям», «что можно обновить».

Что делать:
- Recon-команда выше (сервер автоматически приложит информацию о новых релизах). Никаких upgrade.
- Отчёт пользователю: текущий релиз, последний доступный stable-релиз (если отличается — явно сказать «доступен новый релиз wb-YYMM, обновится через apt upgrade»), N пакетов в upgradable, из них M — wb-\*. Предложи запустить сценарий A.

## Execute: паттерн для длинных apt/wb-release

```
jobId = ssh_exec_async(sn, "<команда>", label="<краткая метка>")
loop:
  status = job_status(sn, jobId)
  tail   = job_tail(sn, jobId, fromLine = lastShown)
  показать пользователю последние строки (если есть новые)
  if status.state == "exited":
    break
  пауза ~10с (не опрашивай в цикле без пауз — это тоже токены)
```

Не забывай `job_cancel(sn, jobId)` если пользователь сказал «отмени» — иначе процесс продолжит идти.

## Verify: после обновления

- `wb-release` — релиз тот, что ожидался.
- `apt list --upgradable` — пусто (или остались только «kept back», про которые пользователь в курсе).
- `systemctl --failed` — нет новых упавших сервисов. Если есть — читай `journalctl -u <unit> -n 50`.
- `diff_snapshot` — покажи пользователю что добавилось/пропало/поменяло версию.
- Для критичных сервисов (`wb-rules`, `wb-mqtt-serial`, `wb-mqtt-confed`): `systemctl is-active` + `journalctl -u <unit> -n 20` — убедись, что стартовали после рестарта.

## Сценарий D: factory reset / откат пакетов через wb-release

Триггеры: «откати пакеты», «после неудачного апгрейда», «контроллер сломался после смены релиза», «верни как было», «factory reset».

`wb-release` имеет два восстановительных режима — это **НЕ** FIT-прошивка (которую мы не запускаем), а пересборка пакетного состояния из репозиториев:

- **`wb-release -p` (`--reset-packages`)** — приводит ВСЕ wb-\* пакеты к версиям текущего релиза. Откатывает downgrade-ом то, что стоит выше; ставит то, что должно быть, но было удалено. Конфиги в `/etc/` НЕ трогает (`dpkg --force-confold` по умолчанию).
- **`wb-release -r` (`--regenerate`)** — перегенерирует системные конфиги из шаблонов wb-configs. Полезно если кто-то напортил в `/etc/network/interfaces` или подобных автоконфигурируемых файлах. Кастомные правки могут быть перезаписаны — обязательно `controller-backup` перед.

Что делать (и `-p`, и `-r` идут одинаково):
1. `controller-backup` — **обязательный**. Оба режима меняют пакеты/конфиги.
2. Покажи пользователю что собираемся делать и почему. Не запускай ради профилактики.
3. **HITL.** Жди явного «да».
4. `ssh_exec_async(sn, "wb-release -y -p", label="wb-release -p")` (или `-r`).
5. `job_tail` до `state: "exited"`. SSH может временно отвалиться — жди, процесс под `systemd-run --collect`.
6. После: `wb-release` (текущее состояние), `apt list --upgradable` (должно быть пусто), `systemctl --failed`.
7. Если перезагрузка не произошла автоматически — предложи `systemctl reboot` с HITL.

**Аппаратный factory reset** (зажать кнопку → заводская прошивка через FIT) — это сервисная операция, мы её не запускаем. Если пакетный откат не помог — пользователь должен сам делать FIT по [wiki](https://wirenboard.com/wiki/Wirenboard_Update).

## Что НЕ делаем

- FIT-прошивку (`wb-fw-update`, `swupdate`, `wb-run-update`) — **запрещено**, см. системный промпт. Если пользователь просит FIT — объясни, что это операция для заводских сервис-кейсов, и откажи.
- `apt upgrade` без бэкапа на production-контроллере с кастомными пакетами/конфигами — пользователь может потерять modified-файлы из-за `dpkg --force-confold/confnew`.
- `apt-get -y purge` ради «почистить» — трогает зависимости. Если нужно удалить один пакет — точечно `apt-get remove <pkg>` с показом зависимостей до.
- Менять релиз «назад» с testing на stable — формально команда та же (`wb-release -t wb-2504`), но обратный путь может оставить пакеты с более высокой версией (downgrade не делается автоматически). Предупреди пользователя: **после отката на stable может понадобиться `wb-release -p` (reset-packages)** — это перекатывает все пакеты на версии релиза, с HITL.

## Грабли

- Запускать сетевые/долгие apt-команды (`apt update`, `apt install`, `apt upgrade`) или `wb-release -t` обычным `ssh_exec` — сервер заблокирует, используй `ssh_exec_async`.
- Игнорировать `apt list --upgradable` и сразу запускать upgrade — не увидишь, например, что обновляется ядро или wb-core, где стоит предупредить пользователя.
- Запустить `wb-release -t` без `-y` — процесс повиснет в ожидании stdin (его никто не подаст).
- Считать что после `wb-release -t` пакеты подтянутся сами вечно — процесс синхронный, если упал посередине, нужно разбираться (`apt-get -f install`, потом `apt upgrade`).
- Не бэкапить перед сменой релиза — если кастомные пакеты (nodered, zigbee2mqtt) нового релиза не потянут старые конфиги, откатиться сложно.
- Перезагружать контроллер вручную (`reboot`) в середине `apt upgrade` — почти гарантированно сломает dpkg. Даже если SSH подвис, процесс apt на контроллере обычно живёт (`systemd-run --collect`), дождись его.
- После смены релиза не проверять `systemctl --failed` — новые пакеты могут не включиться из-за кастомного unit-файла, перекрывающего пакетный; мелкая поломка, легко пропустить.
