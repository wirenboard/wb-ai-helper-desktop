# skill-creator

Как написать короткий полезный скилл и сохранить его в каталог через `create_skill`. Подгружай, когда пользователь просит «изучи X и запомни», «почитай доки и научись», «создай/обнови скилл про Y», «сделай из этого скилл», «удали скилл Z». Скилл — это компактная инструкция, которую ты же (модель) получишь в будущем диалоге через `load_skill("<name>")`: не пересказ документации, а императивная выжимка — как делать правильно, чего не делать, типичные грабли, куда идти за подробностями.

## Когда писать скилл

- Пользователь явно просит: «запомни», «создай/обнови скилл про X», «сделай из этого скилл».
- Тема повторяющаяся (правила, Modbus, обновления) — не разовый вопрос.
- Уже есть конкретика: реальные имена каналов/команд, сообщения об ошибках, URL доков.

Не писать: единичный вопрос, общее рассуждение, пересказ одной статьи.

## Процесс

1. **Проверь каталог** — возможно подходящий скилл уже есть. Если есть и пользователь просит «обнови» — вызывай `create_skill` с тем же `name` (upsert).
2. **Собери конкретику** — прежде чем писать, разберись в теме:
   - `ssh_exec` / `read_file` — посмотри конфиги, пакеты, логи на контроллере
   - `web_fetch` — загрузи документацию с вики/GitHub напрямую
   - `web_search` — поищи в интернете, если протокол/устройство незнакомое (DALI, DMX, KNX)
   - `mqtt_rpc` — узнай текущий конфиг, шаблоны, параметры через RPC драйверов
   - Живой прогон на контроллере — лучший источник правды.
3. **Напиши `content`** по формату ниже. `name` — kebab-case, короткое (`wb-rules`, `controller-backup`).
4. **Покажи пользователю** — особенно лид-абзац (он будет триггером в каталоге), получи подтверждение.
5. **Вызови `create_skill({name, content})`** — скилл сразу появится в каталоге следующих диалогов.

Удаление — `delete_skill({name})`. Только пользовательские; системные защищены.

## Как писать содержимое

Пиши для себя-будущего, уже занятого задачей пользователя. Экономь его (своё) внимание.

- **Императив, не нарратив.** «Используй RPC `wbrules/Editor/Save`» > «обычно используется RPC».
- **Запреты явные.** «НЕ правь `/etc/wb-mqtt-serial.conf` через `write_file`» сильнее любого «рекомендуется».
- **Реальные имена в примерах.** `dev["wb-mr6c_7/K1"]`, не `dev["device/channel"]` — абстрактные плейсхолдеры модель вставит в код буквально.
- **Конкретные строки ошибок.** Знаешь текст из лога — впиши дословно. Модель свяжет причину со следствием.
- **Ссылки через http(s).** Только `web_fetch`-совместимые URL, никаких «см. в коде».
- **Не дублируй системный промпт.** Топология, правила про SN, приоритет web_fetch — всё это уже есть у модели.
- **Объём 30–150 строк.** Длиннее — скорее всего пересказываешь документацию. Разбей на два скилла.

### Указывай тулы для выполнения задачи

Скилл читает будущая модель, которая не знает как действовать. Явно пиши какие тулы использовать для решения задачи по скиллу:

- `write_file(sn, path, content)` — записать скрипт/конфиг на контроллер
- `ssh_exec(sn, command)` — выполнить команду (chmod, запуск, проверка результата)
- `read_file(sn, path)` — прочитать файл
- `mqtt_rpc` / `mqtt_read` / `mqtt_publish` — работа с MQTT и RPC драйверов
- `web_fetch` / `web_search` — документация и поиск

Пример: скилл про кастомный serial-протокол должен содержать «положи скрипт через `write_file(sn, "/mnt/data/scripts/my-proto.py", content)`, сделай `ssh_exec(sn, "chmod +x /mnt/data/scripts/my-proto.py")`, запусти через `ssh_exec`». Не абстрактное «создайте файл на контроллере».

## Структура

Обязательно: `# <name>` первой строкой, пустая строка, лид-абзац 1–3 предложения, затем разделы. **Лид-абзац важнее всего** — сервер вытаскивает его как `description` и показывает в каталоге. Это триггер «при каких формулировках подгружать скилл»: пиши живые формулировки пользователя, а не теги.

Хорошо: «Modbus/RS-485 на контроллере. Доступ через RPC `wb-mqtt-serial/...`. Подгружай на „канал не публикуется", „сканируй шину", „добавь устройство на RS-485".»

Плохо: «Скилл про modbus» / «mqtt, serial, rs485» / «этот скилл описывает…».

```
# <name>

<Лид: что это, ключевая конвенция, живые триггеры. 1-3 предложения.>

## <Главный путь — делай так, а не иначе>

<Реальные команды, реальные имена, конкретные тулы.>

## <Шпаргалка / типовые сценарии>

<Fenced-блоки кода, типовые снипеты.>

## Грабли

- <типичная ошибка и как избежать>

## Документация

- Wiki: <https://wirenboard.com/wiki/...>
- Исходники: <https://github.com/wirenboard/...>
```

Заголовки после лида подбирай под тему. Разделы «Грабли» и «Документация» обязательны.

## Грабли

- Лид-абзац как «оглавление» вместо живых триггеров → модель не поймёт когда подгружать.
- Нет лид-абзаца, сразу `## Заголовок` → `create_skill` вернёт ошибку.
- Написал для человека («этот скилл поможет вам…») → бесполезно, модель читает как инструкцию.
- Абстрактный пример `dev["device/channel"]` → модель вставит в код буквально.
- Не указал тулы → модель будет гадать как выполнять задачу.
- Скилл длиннее 150 строк → вырежи пересказ, замени на ссылку.
- Попытался обновить системный скилл → ошибка, системные защищены.
