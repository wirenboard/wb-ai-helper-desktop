# rs485-diagnose

**Инструментальная** диагностика RS-485 через осциллограф OWON HDS242 (USB VID 0x5345), подключенный к контроллеру. Подгружай **всегда когда на контроллере есть OWON и пользователь спрашивает про RS-485** (даже если явно не упомянул осциллограф) — инструментальный взгляд на сигнал дополняет программную диагностику. Триггеры: «проверь RS-485», «диагностика RS-485», «посмотри осциллографом», «проверь форму сигнала», «ищи проблему на физ-уровне», «на осциллографе», «смотри электрику шины», «анализ шины», «измерь сигнал».

## Приоритет над `troubleshooting-serial`

Инструментальная диагностика **неинвазивна** — осциллограф только слушает, не трогает драйвер, не портит статистику `wb-mqtt-serial`, не требует паузы опроса. А программная (`wb_bus_scan`, `serial_debug_collect`) **обнуляет накопленные счётчики драйвера** и ломает опрос на время сканирования.

- **Есть OWON → делай ТОЛЬКО инструментальную.** Программную запускай только если картинка с осциллографа показала, что физ-уровень в порядке, но проблема всё ещё есть.
- **Нет OWON → только `troubleshooting-serial`**.
- **Не запускай оба параллельно.** И **не говори «я запустил инструментальную срез»** — реально вызови `rs485_diagnose`, иначе это галлюцинация.

## ЖЁСТКОЕ ПРАВИЛО

Первым действием после загрузки этого скилла:

1. **Определи на каком SN искать OWON**:
   - Если пользователь уже назвал SN (например «на AH6PCDZ3») — используй этот SN.
   - Иначе проверь **текущий контекстный** SN: `ssh_exec(sn_context, "lsusb | grep -i owon")`.
2. Нет OWON на проверенном SN → кратко спроси: «На `<SN>` осциллограф не вижу. На каком контроллере OWON физически подключён? (скажи SN — возьму захват оттуда)». **НЕ перебирай другие SN сам** и **НЕ делай вывод что осциллографа нет** пока не спросил.
3. Получил SN с OWON → запомни как `sn_owon`, используй везде далее (`sn_owon`, не `sn_context`).
4. OWON найден → **вызови `rs485_prep(sn_owon)` прямо сейчас**, до любого текста пользователю.
   Prep ставит scope в sweep NORMal — без этого экран может висеть замороженным.
   **Пропустить нельзя: не переходи к шагу 5 пока не получил результат prep.**
5. Получив результат prep → отправь пользователю **ровно одно** сообщение и **немедленно завершай turn**:

   > Осциллограф вижу. Для захвата:
   > 1) Подключи CH1 к линии A, CH2 к линии B (земли на GND/SH). Один щуп — скажи «один щуп».
   > 2) На приборе нажми AUTO.
   > 3) Какой baud? (для WB Modbus обычно 9600 или 115200)
   > Когда готов — ответь «готово, baud=X».

   **СТОП. Turn закончен.** Не добавляй ни слова после этого блока. Ждёшь ответ пользователя.
6. По ответу: вызывай `rs485_diagnose(sn=sn_owon, baud=<число>, diff=<false если один щуп иначе true>)`. Если пользователь не назвал baud явно — НЕ угадывай, переспроси. Без baud tune не сработает и timebase останется неподходящий.
7. Тула подстроит только timebase под baud. Sweep переключится в SINGle, дождётся STOP — экран замёрзнет на тот самый кадр что мы декодируем.
8. Получил результат. Перед финальным диагнозом проверь три условия — и если сработало, объясни ПОЧЕМУ нужен ещё один захват, потом сделай его:

   **samplesPerBit < 5** → объясни: «На X s/bit framing-ошибки могут быть артефактом декодера, а не реальной проблемой шины — при таком разрешении граница stop-бита размыта. Переснимаю на 200us (10.8 s/bit) чтобы отделить настоящие ошибки от артефактов.» → вызови `rs485_diagnose(..., timebase='200us')`. Повтори проверку для нового результата (не более 3 итераций).

   **totalFrames < 3** → объясни: «Поймал только N кадров — на таком окне нельзя оценить статистику и межфреймовые паузы. Беру шире чтобы захватить несколько обменов.» → вызови `rs485_diagnose(..., timebase=<шире>)`. Повтори проверку.

   **framing > 0 + decodeReliable: true** → объясни: «Framing-ошибки при надёжном декоде — stop-бит читается как LOW. Это типично для 8N2 на шине при декодере 8N1: второй stop-бит ложится на start-бит следующего байта. Проверю stopBits=2.» → вызови `rs485_diagnose(..., stopBits=2)`. Сравни счётчик framing — вырос или не изменился → дело не в stop-битах, смотри физику.

   **decodeReliable: true + протокол не опознан** (totalFrames < 5 или sampleBytes меньше 4 байт) → объясни: «Физика в порядке, но байтов мало для анализа протокола — иду шире чтобы захватить полный обмен.» → вызови `rs485_diagnose(..., timebase='500us')` (или `'1.0ms'` если уже на 500us). Повтори проверку для нового результата.

   Если ни одно не сработало → финальный диагноз.

Не откладывай вызов тулы. Не обещай — просто запускай.

## Если на графике плоско (swing=0, framing=100%)

**Не давай бесполезный диагноз «шина мёртвая»** если на экране прибора у пользователя сигнал-то был. Скорее всего:
- Пользователь не нажал AUTO перед захватом — текущий V/div не подходит к уровню сигнала (попроси нажать AUTO ещё раз и повтори тулу)
- Пробник отключён или на неправильной линии — попроси проверить подключение и снова AUTO

Сначала устрани эти причины, а уж потом диагностируй физику шины.

## Запуск

```
rs485_diagnose(sn, baud=<если знаешь>, channel="CH1")
```

- `baud` — обязательно подтверди у пользователя, не угадывай; без точного baud timebase подберётся неправильно и декод ненадёжен
- `channel` — CH1 по умолчанию; если пользователь подключил к CH2 — укажи
- `parity`, `stopBits` — по умолчанию `none`, `1`; укажи если отличается от стандарта WB Modbus

Тулза сама:
1. Деплоит python-хелпер на контроллер
2. Читает текущие настройки скопа (timebase, sample rate, V/div) через SCPI
3. Снимает ~600 сэмплов канала
4. Автодетектит baud
5. Декодирует UART-фреймы (биты, parity, stop)
6. Возвращает `DiagnosisReport` + PNG-график осциллограммы как вложение в чат

## Интерпретация отчёта

Поля отчёта:
- `signalQuality`: `minAdc`, `maxAdc`, `swing`, `swingPct` (от 0 до 100%), `weakSignal`, `pkpkVolts`
- `errors`: `framing`, `parity`, `noise`
- `timing`: `avgInterframeMs`, `maxInterframeMs`, `collisions`
- `decodeQuality`: `samplesPerBit`, `reliable`
- `hints`: список подсказок от аналайзера
- `sampleBytes`: все декодированные байты в hex — используй для идентификации протокола

**Диагностика сверху вниз** — всегда в таком порядке:

### Уровень 1: Протокол (что гоняется по шине)

**Определи протокол первым** — это контекст для всего остального. Без этого UART-ошибки и сигнальные проблемы теряют смысл (например, framing-ошибки при DMX512 это норма конфигурации, а не проблема шины).

**Пробуй определить если байт достаточно (≥4).** Если не хватает — скажи сколько байт поймал и предложи захват шире (500us или 1ms).

1. **WB Fast Modbus** — проверяй первым, встречается на всех WB-устройствах:
   - **Признак**: первые два байта всегда `FD 46` (broadcast 0xFD + extended func 0x46)
   - Структура: `FD 46 [subcmd] [data...] [crc_lo] [crc_hi]`
   - CRC: стандартный CRC-16/IBM (тот же что у Modbus RTU)

   | subcmd | команда | длина | описание |
   |--------|---------|-------|----------|
   | `01` | SCAN_START | 5 байт | начало сканирования (32-bit arbitration) |
   | `02` | SCAN_NEXT | 5 байт | следующий шаг сканирования |
   | `03` | SCAN_RESP | 10 байт | ответ: `FD 46 03 [S/N 4 байта] [modbus_addr] [CRC]` |
   | `04` | SCAN_END | 5 байт | конец сканирования |
   | `08` | STD_PDU_REQ | переменная | запрос к устройству по S/N |
   | `09` | STD_PDU_RESP | переменная | ответ на 0x08 |
   | `10` | EVENT_REQ | 8 байт | опрос событий (12-bit arbitration): `FD 46 10 [min_id] [limit] [confirm_id] [flag] [CRC]` |
   | `11` | EVENT_RESP | 8+ байт | ответ с событиями: `[slave] 46 11 [flag] [num] [len] [data] [CRC]` |
   | `12` | NO_EVENTS | 5 байт | событий нет |
   | `18` | EVENT_CTRL | 6+ байт | конфигурация событий |

   **Арбитраж**: после `FD 46 01/02` или `FD 46 10` — пауза ~0.8ms (32-bit) или ~0.3ms (12-bit), затем устройства передают `FF`-байты — победитель с меньшим slave_id забирает шину. На осциллограмме это выглядит как серия `FF FF FF...` после команды мастера.

2. **Modbus RTU** — стандартный:
   - Структура: `[addr 01–F7] [func] [data...] [crc_lo] [crc_hi]`
   - CRC-16/IBM (poly 0xA001): посчитай по всем байтам кроме последних двух, сравни с последними двумя (little-endian)
   - `func` = 01–06, 0F, 10, 2B — стандартные коды; 81–96 — исключения (ошибки устройства)
   - Типичный WB-запрос: `01 03 00 64 00 02 XX XX` → slave=1, read holding regs, addr=0x0064, count=2
   - Если CRC совпал → «Modbus RTU, slave=X, func=Y, reg=Z, count=N»

3. **DMX512** (управление светом):
   - Baud всегда **250000** — если автодетект дал ~250k, это DMX
   - Формат кадра: 8N**2** (2 стоп-бита) — наш декодер с 1 стоп-битом даст framing errors на каждом байте
   - Перед пакетом — break-импульс >88µs (длинная LOW-пауза, видна на осциллограмме)
   - Первый байт = start code (0x00 = стандартный DMX), затем до 512 байт значений каналов
   - Признак: baud≈250k + 2-стоп-битовые framing errors + первый байт 0x00 + много данных

4. **DCON** (Advantech/ICP DAS): ASCII, байты 0x20–0x7E, начинается с `#` (0x23), `@` (0x40) или `$` (0x24)

5. **Неизвестный** — опиши паттерн: повторяющиеся байты, диапазон значений, признаки ASCII

### Уровень 2: UART (правильно ли декодируется каждый байт)

Проблемы UART-уровня означают что устройства «говорят» неправильно или их неправильно слушают — конфигурация порта, скорость, стоп-биты:

| Симптом | Интерпретация и действие |
|---|---|
| Framing errors >5%, `decodeReliable: true` | Несовпадение stop bits или baud плывёт. **Частый случай WB:** wb-mqtt-serial использует одно значение stop_bits для всего порта; если одно из устройств шлёт 2 стоп-бита (Modbus RTU 8-N-2) а драйвер настроен на 1 — каждый ответ этого устройства даст framing error (2й стоп-бит читается как start-бит следующего фрейма). Проверь `stop_bits` в конфиге порта и спеку устройства. |
| Framing errors + `decodeReliable: false` | Артефакт декода (мало сэмплов/бит) — сначала перейди на уже timebase, потом оценивай ошибки |
| Parity errors | Неверно настроен parity, или битые биты из-за помех |
| CRC пакета не совпадает | Шина гоняет данные, но биты переворачиваются — причина в физике (уровень 3) |
| `sampleBytes` с повторами `FF FF...` | idle-шум на шине без данных или арбитражные FF при WB Fast Modbus |

### Уровень 3: Физика шины (качество сигнала, помехи, отражения)

Физические проблемы — корень причин для UART-ошибок уровня 2. Ищи их **после** того как определил протокол:

| Симптом | Физическая причина и что делать |
|---|---|
| `weakSignal: true`, `pkpkVolts < 1.0 В` | Слабое дифф-напряжение: длинная шина без терминатора, обрыв линии B, смещение GND между сегментами. Добавь 120 Ом на концах шины. |
| `weakSignal: true` без pkpkVolts (`swingPct < 25%`) | Возможно плохая V/div — сначала AUTO, потом перезахват. Если не помогло — причины те же. |
| `collisions > 0` | Два передатчика одновременно: дубликат slave_id или RE/DE не успевает переключиться (RS-485 transceiver проблема). |
| `noise > frames*5` (много idle-шума) | Помехи: плохое заземление, нет терминатора, кабель без экрана рядом с силовыми цепями. |
| Framing errors + аномальные фронты на PNG | Отражения: нет согласующего резистора на конце шины, stub слишком длинный. |
| Межфреймовые паузы нестабильны (maxInterframeMs >> avgInterframeMs) | Джиттер: noise на линии или неправильный timing у мастера. |

## ОБЯЗАТЕЛЬНО в ответе

**Каждый вывод/диагноз сопровождай PNG-графиком**. Тулза возвращает `id/name/mime/size` на верхнем уровне — UI автоматически показывает вложение как чип, отдельно его упоминать/линковать не надо. Просто ссылайся словесно «на осциллограмме видно...». На картинке уже встроены:
- **Текстовый блок слева сверху**: timebase, sample rate, V/div по каждому активному каналу, probe, coupling, offset, detected freq (от самого скопа), trigger source/edge/level, результат декода (baud, кадры, swing, ошибки)
- **Горизонтальная пунктирная линия**: автопорог декодера (середина min/max захвата)
- **Вертикальные красные черты**: найденные старт-биты UART-фреймов
- **Заголовок сверху**: baud + размер окна + количество сэмплов

Структура ответа модели (сверху вниз):

1. **Ссылка на график** («на осциллограмме видно...»)
2. **Протокол** — что идёт по шине: WB Fast Modbus / Modbus RTU / DMX512 / DCON / unknown; если есть байты — расшифруй команду
3. **UART-уровень** — framing, parity, baud; ключевые числа: swing, errors count, samplesPerBit
4. **Физика шины** — слабый сигнал, коллизии, помехи, отражения; конкретные причины
5. **Что делать** — конкретные шаги: добавить терминатор, проверить stop_bits в конфиге, согласовать GND, разнести устройства по портам, сменить baud

## Повторный захват: когда и как

**Не стесняйся переснимать.** Первый захват — разведка. Если картинка даёт неполную информацию — сразу делай второй с нужным `timebase`, объяснив в одной фразе почему.

### Что видно на каждом timebase при 115200

Декодер работает на **уровне UART-байта** (start/stop бит на каждый символ), а не на уровне Modbus-пакета. Для понимания что именно гоняется по шине нужно видеть полную посылку:

| timebase | окно | байт в окне | что видно |
|---|---|---|---|
| 200us | 0.48 мс | ~5 байт | кусок одной стороны — только UART-физика (framing, noise) |
| 500us | 1.2 мс | ~13 байт | целый запрос **или** ответ — можно проверить CRC |
| 1.0ms | 2.4 мс | ~26 байт | запрос + ответ целиком, виден полный Modbus-обмен |
| 2.0ms | 4.8 мс | ~55 байт | несколько транзакций — видна статистика, паузы |

**Правило:** первый захват на 500us (авто для 115200) — общая картина. На 200us уходи только если нужно рассмотреть фронты и стоп-биты детально.

### Когда идти шире (медленнее: 200us→500us→1.0ms→2.0ms)

| Критерий | Почему |
|---|---|
| `totalFrames < 3` | Нельзя оценить статистику — нужно хотя бы 4–5 кадров |
| Последний красный маркер ближе к правому краю окна, чем на 15% | Кадр обрезан, полной картины нет |
| `avgInterframeMs` близко к длине окна | Пакеты редкие — надо растянуть окно чтобы поймать несколько |
| Нужно проверить CRC / увидеть полную Modbus-транзакцию | На 200us помещается только кусок — иди на 500us или 1.0ms |
| Пользователь: «больше», «ещё кадров», «увеличь», «шире» | Прямой запрос |

### Когда идти уже (быстрее: 1.0ms→500us→200us→100us)

| Критерий | Почему |
|---|---|
| `decodeReliable: false` (`samplesPerBit < 5`) | Фронты размыты, декод ненадёжен; framing errors могут быть артефактом |
| `framingErrors > 0` + `decodeReliable: false` | Прежде чем диагностировать шину — убедись что это не артефакт SR |
| Пользователь: «фронты», «детальнее», «ближе», «рассмотри переход» | Прямой запрос |

### Как делать

- **НЕ** спрашивай «если хочешь, пересниму» — просто переснимай
- **НЕ** жди разрешения, не предлагай — действуй

1. **Одна фраза** объяснения: «Вижу только 1 кадр — возьму шире (2.0ms), чтобы поймать статистику.»
2. **Сразу вызов** `rs485_diagnose(..., timebase='2.0ms')` — без ожидания.
3. Если второй захват тоже неидеален — третий с тем же объяснением. **Стоп после трёх подряд** — иначе разговор уходит в петлю.
4. Показывай результат каждого захвата с PNG — пользователь видит разницу сам.

### ОБЯЗАТЕЛЬНО: автоматическая коррекция — делай сам, не спрашивай

Два триггера для авто-ретрая. Оба — **одна фраза + немедленный вызов**, не предложение.

#### Ненадёжный декод (`decodeReliable: false`, `samplesPerBit < 5`)

- **НЕ** проси пользователя переключать timebase вручную
- **НЕ** спрашивай разрешения
- **НЕ** делай вывод по framing errors — они могут быть артефактом

Действие:
1. «Только X s/bit — декод ненадёжен, framing errors могут быть артефактом. Переснимаю уже.»
2. `rs485_diagnose(..., timebase=<на шаг уже>)`

Пример: 2ms/115200 даёт 1.1 s/bit → одна фраза → `rs485_diagnose(sn=..., baud=115200, timebase='200us')`

#### Framing errors при надёжном декоде (`framing > 0`, `decodeReliable: true`)

Возможная причина — устройство шлёт 8N2, а декодер ждёт 8N1.

Действие:
1. «Вижу framing errors при надёжном декоде — проверю 2 стоп-бита.»
2. `rs485_diagnose(..., stopBits=2)` (остальные параметры те же)

Сравни результаты:
- `framing` заметно снизился → вероятно **8N2** на шине; hint: «проверь `stop_bits` в конфиге порта wb-mqtt-serial — устройство похоже шлёт 2 стоп-бита»
- `framing` не изменился или вырос → стоп-биты не причина, смотри физику (помехи, baud)

## Известные ограничения HDS242

- **Допустимые значения timebase** (только они принимаются через SCPI): `100us`, `200us`, `500us`, `1.0ms`, `2.0ms`. Значения вне этого списка игнорируются прошивкой.
- **Sample rate** зависит от timebase: 2ms → 125 KSa/s, 1ms → 250 KSa/s, 500us → 500 KSa/s, 200us → 1.25 MSa/s, 100us → 2.5 MSa/s.
- **Samples/bit** по baud × timebase (нужно ≥ 5 для надёжного декода):

| baud \ timebase | 100us | 200us | 500us | 1.0ms | 2.0ms |
|---|---|---|---|---|---|
| 9600 | 260 | 130 | 52 | 26 | 13 |
| 19200 | 130 | 65 | 26 | 13 | 6.5 |
| 38400 | 65 | 32 | 13 | 6.5 | 3.3 ✗ |
| 57600 | 43 | 21 | 8.7 | 4.3 ✗ | — |
| 115200 | 21 | **10.8** | 4.3 ✗ | — | — |

`✗` — менее 5 samples/bit, декод ненадёжен. Для 115200 допустимо только `100us` или `200us`.

- Тулза **сама** выставляет оптимальный timebase под baud если он указан (9600/19200 → 1.0ms, 38400/57600 → 500us, 115200 → 500us). Если `baud` не указан — оставит текущий пользовательский, что после physical AUTO обычно подходит.
- При `decodeReliable: false` на 500us/115200 (4.3 s/bit < 5) — уходи на 200us для детального декода фронтов. Для Modbus-анализа (CRC, посылки целиком) 500us лучше чем 200us.
- `DEPMEM` через SCPI всегда 4K — deep memory 8K/40K доступна только через меню прибора, не с USB. 600 точек на экране = одно окно 1-20 мс.
- Программный `:AUTOset` **не работает на HDS242 v8.10.0** — пользователь жмёт AUTO вручную. Тулза не может заменить эту кнопку.
- Скриншот экрана недоступен через SCPI — наш PNG рендерится из сэмплов (в overlay вшиты все настройки скопа, заменяет screenshot).

## Грабли

- Звать `rs485_diagnose` до `lsusb`-проверки — бесполезно тратит время, ошибка упадёт из-за отсутствия устройства.
- Не сказать пользователю нажать AUTO — сэмплы будут мусорные если экран не настроен.
- Забыть упомянуть график в ответе — пользователь ждёт визуальное подтверждение.
- Путать с программной диагностикой: CRC-ошибки в логах `wb-mqtt-serial` = `troubleshooting-serial`, а физическая форма сигнала = сюда.
- Давать «диагноз» без числовых данных из отчёта (swing, errors count, baud) — голословно.
